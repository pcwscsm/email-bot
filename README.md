\# Email Aggregator Bot



> 一个把"邮箱"+"学校就业网"+"未来其他信息源"统一抽象、定时轮询、用 LLM 摘要、推送到飞书群的个人信息聚合工具。

\## 为什么做这个



我每天面对的真实痛点是这样的：



\- 三个邮箱（QQ、Gmail、学校 BJTU 邮箱）分别要打开看，\*\*经常漏看学校通知\*\*——尤其是教务处、就业指导中心发的截止日期类邮件

\- 学校就业网首页的实习信息每天都更新，但列表是 JS 渲染、看一眼要点好几次\*\*

\- 这些信息分散在不同地方，\*\*我没有一个"今天要看的东西"的统一入口\*\*



直接用现成方案的问题：



\- 邮件转发：太啰嗦，每条都推会刷屏

\- RSS 阅读器：不支持邮件，也不支持就业网这种 JS 渲染站

\- 微信公众号订阅：根本没有学校就业网的公众号



所以我自己写了一个。\*\*核心目标不是"覆盖一切信息"，而是"覆盖能自动化的部分，避免错过该看的"\*\*。



\---



\## 它做什么



每 10 分钟（可调）跑一次：



1\. \*\*拉取\*\*——分别从三个邮箱、就业网两个 API 拉新数据

2\. \*\*过滤\*\*——按硬规则筛掉明显不相关的（地区黑名单、过期宣讲会等）

3\. \*\*理解\*\*——用 LLM 给每条信息生成摘要 + 重要性等级（高/中/低）

4\. \*\*推送\*\*——把"高/中"重要性的合并成一张飞书卡片推到群里



每天大概推 1-2 张卡片，每张卡片 5-10 条信息。\*\*信号密度高、噪声极低\*\*。



下面是飞书卡片的真实形态（示意）：



```

┌─ 邮件 \& 就业速递   5 条新消息 · 14:30 ─┐

│

│ 🔴 高重要 (1 条)

│   关于2026春季学期选课工作的通知

│   发件: 教务处 · BJTU

│   摘要: 5月10日前完成选课，逾期不补

│   重要性高·教务通知 · 2026-05-04 09:12

│

│ 🟡 中重要 (4 条)

│   北京顺丰同城 · 北京海淀 · 本科/硕士

│   来源: BJTU实习

│   摘要: 6 个实习岗位，对本科开放

│   不限专业·北京可达 · 截止 2026-07-30

│   ...

```

\## 架构



整个系统围绕一个\*\*统一的 Source 抽象\*\*展开。这是这个项目最值得一讲的设计决策。



```

┌─────────────────────────────────────────────────────────────┐

│                          main()                             │

│                定时调度入口（每 10 分钟一次）                  │

└────────┬────────────────────────────────────────────────────┘

&#x20;        │

&#x20;        │  build\_sources(config) → list\[Source]

&#x20;        ▼

┌─────────────────────────────────────────────────────────────┐

│                   Source (抽象基类)                           │

│  契约: fetch(state\_slice) → (items, state\_updates)           │

│       state\_keys() → list\[str]                               │

│  约束: 纯函数 / 不写盘 / 失败时不污染 state                     │

└────────┬────────────────────────────────────────────────────┘

&#x20;        │

&#x20;        │  三个具体实现

&#x20;        ▼

┌─────────────────┬─────────────────┬─────────────────────────┐

│   IMAPSource    │   WebSource     │   JSONAPISource         │

│   (邮件)         │   (HTML 爬虫)   │   (JSON 接口)            │

│                 │                 │                          │

│  · QQ           │  · 学院通知      │  ├─ BJTUInternshipSource │

│  · Gmail        │   (CSS 选择器)   │  └─ BJTUFairSource       │

│  · BJTU         │  · 配置驱动      │                          │

│                 │                 │  · 字段映射 / 硬过滤        │

│  · 增量 UID     │  · seen\_ids     │  · seen\_ids               │

│  · 多文件夹     │   FIFO 容量管理   │   FIFO 容量管理            │

└─────────────────┴─────────────────┴─────────────────────────┘

&#x20;        │

&#x20;        ▼  统一的 Item 字典（source\_type / source\_name / item\_id /

&#x20;           title / body / url / date / route / ...）

&#x20;        │

┌─────────────────────────────────────────────────────────────┐

│                    summarize\_item()                         │

│   按 source\_type 选不同提示词模板，LLM 输出摘要 + 重要性 + 理由  │

└────────┬────────────────────────────────────────────────────┘

&#x20;        │

&#x20;        ▼

┌─────────────────────────────────────────────────────────────┐

│                    push\_to\_feishu()                         │

│      按重要性分组，合并成一张交互卡片，推送到机器人 webhook       │

└─────────────────────────────────────────────────────────────┘

&#x20;        │

&#x20;        ▼

&#x20;  state.json (持久化已处理的 UID / 已见过的 ID 列表)

```



\### 一些关键设计决策



\*\*为什么 Source 要做成抽象基类，而不是直接写三个独立函数？\*\*



因为我观察到三件事是所有信息源都要做的：①拉取、②与上次状态对比找新的、③更新状态。

不同的只有"怎么拉""怎么对比"。把共性抽出来之后，加新源只要继承基类、实现 `\_fetch\_records` 这种几行代码的方法就行——比如未来加 RSS 源，\*\*预计代码量 80-120 行，零改动到主流程\*\*。



\*\*为什么用"幂等的纯函数式"接口（fetch 不能写盘）？\*\*



方便测试。这就是这个项目能写 16 个测试用例的根本原因——所有 source 都不依赖外部状态、不产生副作用，可以在 mock 数据上反复跑、断言输出。\*\*这是大学课程教不到、但工业界研发流程的核心做法\*\*。



\*\*为什么状态文件用 `source\_type:source\_name:folder` 这种三段 key？\*\*



防止不同 source 的 state 互相覆盖。`imap:QQ:INBOX` 和 `bjtu\_internship:BJTU实习` 在同一个文件里和谐共存，未来加任何新 source 都不会命名冲突。



\*\*为什么 LLM 提示词要按 source\_type 分化？\*\*



"重要性"在不同上下文里含义完全不同——

\- 邮件的"高"= 学校官方/截止日期/账户安全

\- 学院通知的"高"= 选课/学籍/计算机大类相关

\- 实习信息的"高"= 互联网科技公司/北京可达/对本科开放



用同一组提示词强行套，LLM 判断质量会显著下降。

\## 配置



所有配置在 `%USERPROFILE%\\EmailBot\\config.json`（首次运行会自动生成模板）。



最小可用的 config 长这样：



```json

{

&#x20;   "accounts": \[

&#x20;       {

&#x20;           "name":         "QQ",

&#x20;           "host":         "imap.qq.com",

&#x20;           "port":         993,

&#x20;           "user":         "your\_qq@qq.com",

&#x20;           "password":     "16\_digit\_authcode",

&#x20;           "webmail\_url":  "https://mail.qq.com/",

&#x20;           "folders":      \["INBOX"],

&#x20;           "initial\_mode": "recent24h"

&#x20;       }

&#x20;   ],

&#x20;   "websites": \[],

&#x20;   "jsonapi\_sources": \[

&#x20;       {

&#x20;           "name":         "BJTU实习",

&#x20;           "type":         "bjtu\_internship",

&#x20;           "max\_per\_run":  8,

&#x20;           "first\_run\_preview": 3,

&#x20;           "skip\_cities":  \[],

&#x20;           "skip\_natures": \[],

&#x20;           "route":        "mail"

&#x20;       }

&#x20;   ],

&#x20;   "llm": {

&#x20;       "dashscope\_api\_key": "sk-xxxxx",

&#x20;       "model": "qwen-turbo"

&#x20;   },

&#x20;   "feishu": {

&#x20;       "webhook": "https://open.feishu.cn/open-apis/bot/v2/hook/xxxxx"

&#x20;   },

&#x20;   "filters": {

&#x20;       "skip\_low\_importance": true,

&#x20;       "skip\_senders": \["noreply@example.com"],

&#x20;       "max\_per\_run": 20

&#x20;   }

}

```



\### `accounts\[].initial\_mode`：第一次运行时怎么处理历史邮件



| 模式 | 行为 |

|---|---|

| `recent24h` | 拉最近 24 小时的邮件，下次只拉新邮件（默认） |

| `unseen` | 拉所有未读邮件，下次只拉新邮件（适合学校邮箱有大量历史未读） |

| `uid\_only` | 不拉任何邮件，仅记录当前最大 UID 作为基线（适合"我不想看历史，从下一封新邮件开始关注"） |



\### `jsonapi\_sources\[].first\_run\_preview`



JSON API 源首次运行时推送多少条预览。设 `0` 表示完全静默（推荐稳定使用），设 `3` 表示推 3 条让你看看效果（推荐首次接入时）。



\### 命令行选项



```powershell

python email\_bot.py                   # 正常运行一次

python email\_bot.py --dry-run         # 拉取+摘要但不推送（调试用）

python email\_bot.py --setup           # 仅生成 config 模板

python email\_bot.py --list-folders    # 打印每个邮箱的 IMAP 文件夹

python email\_bot.py --reset-state     # 清空 state.json

python email\_bot.py --install-task    # 注册 Windows 任务计划程序（每 10 分钟）

python email\_bot.py --uninstall-task  # 移除该计划任务

```

\## 部署



完整部署流程：



```powershell

\# 1. 安装依赖

pip install requests beautifulsoup4



\# 2. 首次运行生成 config 模板

python email\_bot.py --setup



\# 3. 编辑 %USERPROFILE%\\EmailBot\\config.json 填入真实凭证



\# 4. dry-run 验证一切正常

python email\_bot.py --dry-run



\# 5. 正式跑一次

python email\_bot.py



\# 6. 注册定时任务（每 10 分钟）

python email\_bot.py --install-task

```



部署在本地 Windows 而不是 GitHub Actions / 云服务器的原因：邮箱密码、Token、Cookie 等敏感凭证留在本机最安全。本人电脑每天开机时间足够长，不需要 7×24 服务器。

\## 已知限制



诚实记录这个项目\*\*做不到\*\*或\*\*没做完美\*\*的地方：



\### 设计层面



\- \*\*不抓微信公众号\*\*——腾讯生态对爬虫封闭，所有方案要么违反协议要么不稳定，作为学生项目不值得做。微信通知保持手动看

\- \*\*不抓 ASP.NET ViewState 类页面\*\*——很多老高校官网用 ViewState 渲染，需要逆向 POST 表单，复杂度过高、收益过低

\- \*\*就业网详情页要登录\*\*——飞书卡片附的"详情"链接需要先在浏览器登录就业网才能打开，这是网站设计，不是 bug

\- \*\*学院通知信息密度过低\*\*——曾经实现了对应的 WebSource 但没接入，因为信号太弱（详见 `git log` 里阶段 C 的设计决策）



\### 实现层面



\- \*\*无并发\*\*：`max\_workers=1` 是有意的，因为 QQ 邮箱有频率限制踩过坑。后果：跑一轮约 30 秒，对 10 分钟一次的调度完全够用

\- \*\*无日志轮转\*\*：`bot.log` 会一直增长，需要定期手动清。FIXME 待补

\- \*\*LLM 失败降级简单\*\*：当前是"调用失败就返回 `(标题前 30 字, 中, LLM 调用失败)`"的简单兜底。没做"主备双模型"切换。需要时添加

\- \*\*state.json 单文件\*\*：如果状态损坏会全部丢失。可以靠 `--reset-state` 恢复，但会重新建立基线

\- \*\*第一次跑学校邮箱可能拉不到历史邮件\*\*：因为 BJTU 邮箱用 Coremail 系统，UID 是基于时间戳的大数。`initial\_mode: unseen` 是更可靠的选择



\### 工程层面



\- \*\*无 CI\*\*：测试套要本机手动跑（`python test\_jsonapi.py` 等）

\- \*\*无类型注解\*\*：代码全是 dict 在传，重构时容易出错。修过一次 `\_meta` 字段缺失的 bug

\- \*\*配置无 schema 校验\*\*：填错字段名只会"看起来正常但行为奇怪"，没有启动时报错

\## 项目演化历史



完整的版本演进，每一版都有明确的"做什么、不做什么"边界：



| 版本 | 时间 | 增量 |

|---|---|---|

| v1.0 | 4 月 | 三个邮箱 + IMAP 增量拉取 + LLM 摘要 + 飞书推送 |

| v1.1 | 5 月初 | `initial\_mode` 多模式 + 多文件夹 + state.json v2 自动迁移 |

| v1.2 | 5 月初 | 引入 Source 抽象基类，IMAP 重构为 IMAPSource 类（金标准测试通过） |

| v1.3 | 5 月初 | 加 WebSource (HTML 爬虫)，引入提示词分化 |

| v1.4 | 5 月初 | 加 JSONAPISource + BJTU 实习/宣讲会两个具体源 |



\*\*故意没做的事\*\*（每一项都做过设计但放弃）：



\- 学院官网通知爬取——做完代码发现信噪比太低，弃用（保留代码作为模板）

\- 双群路由——每天就业信息只有 1-2 条，混在邮件群完全可以接受，过早抽象

\- 本地 Ollama 部署——大模型本地部署在我笔记本上效果差成本高，不如付费 API（DashScope）

\- 网页 UI 配置面板——完全没必要，JSON 配置已经够清晰

\## 致谢与心得



这个项目本身在做"工程师早期能力训练"。值得记录的几条心得：



1\. \*\*"识别该弃就弃"比"持续推进"更难学\*\*。学院通知爬虫做完代码再放弃，比一开始不做更有价值——你只有做过才知道不该做

2\. \*\*测试不只是"我没写错"的保险，更是"我可以放心重构"的解锁条件\*\*。每次重构前先写金标准测试，已经成习惯

3\. \*\*抽象不要超前\*\*。Source 基类是写到第二个具体类（WebSource）时才提出来的，不是一开始就设计好的。\*\*抽象是被需求逼出来的，不是想出来的\*\*

4\. \*\*配置外置 + 默认值兜底\*\*。所有可能变的东西放 config，所有 config 字段都有合理默认值

5\. \*\*失败要"安静"\*\*。某个 source 挂了不影响其他 source；网络错误返回空数据而不是抛异常；状态文件损坏自动重置——\*\*让小问题不变成大问题\*\*
